# Day 13 — 性能优化：Action 缓存与模型路由

## 今日改动

实现 LLM 决策结果缓存体系和缓存管理 API，使相同页面的重复执行跳过 LLM 调用，并将缓存统计接入运营监控：

1. **Action 缓存引擎**：`enterprise/llm/action_cache.py` 实现了完整的缓存生命周期——缓存 key 由三部分组成：`action_cache:{org_id}:{dom_hash}:{goal_hash}`，其中 `dom_hash` 是页面 DOM 结构的 SHA-256 哈希（剥除动态内容后计算），`goal_hash` 是导航目标文本的 MD5 哈希。DOM 动态内容剥除规则覆盖 6 类噪声：含长数字的 ID、React 内部 ID（data-reactid）、测试 ID（data-testid）、行内样式（style）、类名（class）、HTML 注释。剥除后统一空白字符归一化，确保结构等价的 DOM 产生相同哈希

2. **ActionCacheStore**：内存缓存存储实现（生产环境替换为 Redis），支持 TTL 过期（默认 24 小时）、按前缀批量清除、过期条目清理、全量清除。内置命中/未命中计数器，实时计算缓存命中率

3. **高级缓存接口**：`cache_action_decision()` 将 LLM 决策（选择了哪个元素、操作类型、决策理由）写入缓存；`lookup_cached_decision()` 在执行前查询缓存，命中时直接返回决策结果跳过 LLM 调用。两个函数自动处理 DOM 哈希和目标哈希的计算，调用方只需传入原始 HTML 和目标文本

4. **缓存管理 API**：`enterprise/llm/cache_routes.py` 提供 5 个管理端点（均要求 admin 权限）——`GET /stats`（缓存统计：条目数/命中数/未命中数/命中率）、`DELETE /task/{task_id}`（按任务清除缓存）、`DELETE /expired`（清除过期条目）、`DELETE /all`（全量清除）、`POST /reset-stats`（重置计数器）

5. **路由注册**：在 `skyvern/forge/api_app.py` 中注册缓存管理路由

新增测试 35 个（Day 13），累计 502 个，全部通过。模型路由已在 Day 9 实现（`enterprise/llm/model_router.py`），Day 13 的缓存统计与 Day 11 的成本估算端点已通过 `model_calls` 数据结构中的 `cache_hit` 字段对接。

## 设计决策

### DOM 哈希的动态内容剥除

缓存命中的关键在于对"页面结构是否相同"的正确判断。完全相同的页面在不同时间渲染时，React/Vue 的内部 ID、行内样式、class 顺序可能不同，但页面的结构和交互元素不变。剥除这 6 类动态属性后再哈希，可以让结构等价的页面命中同一缓存条目。

考虑过的替代方案：
- **只哈希标签结构**（去除所有属性）：过于激进，两个结构相同但交互元素不同的页面（如不同的表单）会被误判为相同
- **哈希完整 DOM**：过于保守，任何 class 或 style 变化都会导致缓存失效，命中率极低
- **当前方案**（保留标签和核心属性如 type/name/href，去除装饰性属性）：在命中率和安全性之间取得平衡

### 24 小时 TTL 的选择

LLM 决策缓存不同于 Dashboard 统计缓存（60 秒 TTL），原因在于：
- 页面结构通常以天为单位变化（网银/保险系统不会频繁改版），24 小时 TTL 可以覆盖一个完整工作日的重复任务执行
- TTL 过短（如 1 小时）会导致每天开始工作时大量缓存冷启动，无法发挥复用效果
- TTL 过长（如 7 天）在页面升级后会导致缓存中的决策与新页面不匹配，需要手动清除
- DOM 哈希本身提供了一道安全屏障——如果页面结构真的变了，哈希不匹配，自然不会命中旧缓存

### 模块级单例 + configure 注入

`ActionCacheStore` 使用模块级单例（`_cache_store`），通过 `configure_cache_store()` 替换，与 Day 10 的 `set_key()` 和 Day 11 的 `configure_stores()` 保持一致的测试注入模式。生产环境中该单例将替换为 Redis 后端实现，接口不变。

## 金融企业场景下的工程意义

**重复任务的成本优化**：银行网银账单采集、保险理赔状态查询等工作流每天执行数十到数百次，目标页面结构完全相同。Action 缓存使这些重复执行跳过 LLM 调用，直接复用之前的决策结果。按每次 LLM 调用约 $0.01-0.05 计算，一个拥有 100 条重复任务的组织每天可节省 $1-5，月度节省 $30-150。这些数据通过缓存统计 API 可被运营大屏实时展示。

**执行速度提升**：LLM 调用延迟通常在 1-5 秒，缓存命中时延迟降为亚毫秒级，对包含多步操作的工作流（如 5 步网银操作）可将总执行时间从 25-30 秒缩短到 5-10 秒。这对高频批量任务（如每日自动对账）的执行效率提升显著。

**缓存管理的运维可控性**：admin 可以通过 API 查看缓存命中率——如果命中率持续低于 50%，可能意味着 DOM 剥除规则需要调整（过多动态内容未被正确剥除）。手动清除缓存的能力在页面升级后特别重要，避免 RPA 使用过时的决策结果操作新版页面。

## 踩坑记录

1. **HTML 注释剥除后的残留空格**：最初将所有动态内容统一替换为空格（`pat.sub(" ", text)`），导致 `<div><!-- comment --><p>` 变成 `<div> <p>`，与 `<div><p>` 的哈希不同。修复方案是将 HTML 注释单独处理，替换为空字符串（`""`）而非空格，其他属性仍替换为空格（因为属性移除后需要分隔相邻标记）

2. **Bash shell 对 `!` 的转义**：在 bash 交互式调试时，`<!-- comment -->` 中的 `!` 被 bash 的历史替换功能解释为上一条命令引用，导致 `conda run python -c` 中的字符串被错误修改。解决方法是使用 `chr(33)` 构造字符或写入独立脚本文件，而非在命令行中直接使用含 `!` 的字符串
