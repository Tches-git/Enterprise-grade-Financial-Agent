# Day 8 — 全链路审计与合规存储

## 今日改动

完成全链路审计日志系统，覆盖每步操作的前后截图、输入脱敏、审批关联和多维度查询，核心工作分五部分：

1. **审计日志数据模型**：`enterprise/audit/models.py` 定义了 `AuditLogModel`，记录 RPA 任务每步 action 的完整上下文——任务/组织/部门/业务线关联、action 序号与类型（10 种，涵盖 click/input_text/navigate 等）、目标元素描述、脱敏后的输入值及原始值 SHA-256 哈希、页面 URL、操作前后截图 MinIO object key、执行时长（毫秒）、执行人（agent 或 user_id）、审批状态与审批人、执行结果与错误信息
2. **输入值脱敏引擎**：`enterprise/audit/sanitizer.py` 实现了 4 条默认脱敏规则——银行卡号仅保留末 4 位（支持空格/短横线分隔格式）、密码类字段完全掩码（兼容 password/passwd/pwd/密码/口令多种关键词）、身份证号仅保留末 4 位（支持末位 X）、手机号中间 4 位掩码。金额信息不脱敏（业务关键数据）。同时提供 `hash_raw_value()` 生成原始输入的 SHA-256 哈希，供合规审计时的完整性校验
3. **MinIO 截图存储**：`enterprise/audit/storage.py` 实现了结构化的对象存储方案——object key 格式为 `audit/{org_id}/{task_id}/{序号}_{before|after}_{uuid}.png`，bucket 按月自动创建（`finrpa-audit-{YYYYMM}`），支持生成 1 小时有效期的预签名 URL
4. **审计日志写入器**：`enterprise/audit/logger.py` 封装了 `write_audit_log()` 异步函数，自动调用脱敏引擎处理输入值并持久化到数据库。写入失败时记录 `AUDIT_LOG_FAILURE` 系统告警但不抛出异常，确保审计故障不影响 RPA 主流程
5. **审计查询 API**：`enterprise/audit/routes.py` 提供 `GET /enterprise/audit/logs` 接口，支持按 task_id、action_type、executor、时间范围多维度筛选，结果分页（默认 20 条/页，最大 100 条），截图字段返回预签名临时 URL

在 `skyvern/forge/api_app.py` 中注册审计路由。新增测试 44 个（Day 8），累计 345 个，全部通过。

## 设计决策

### 脱敏后存储 + 哈希校验的双重策略

审计日志中的输入值面临矛盾需求：合规要求完整记录操作轨迹，但信息安全要求敏感数据不能明文存储。解决方案：

- `input_value` 字段存储脱敏后的值（如 `****1234`），满足操作回溯需求——审计人员可以看到"操作了某张以 1234 结尾的卡"
- `input_value_raw_hash` 字段存储原始值的 SHA-256 哈希，满足完整性校验需求——如果调查时需要验证某个特定卡号是否与审计记录匹配，可以计算该卡号的哈希并比对

这种方案在《商业银行信息科技风险管理指引》要求的"操作留痕"和《个人信息保护法》要求的"最小必要"之间取得了平衡。

### 按月分 Bucket 而非按组织分

MinIO bucket 按月创建（`finrpa-audit-202603`）而非按组织创建（`finrpa-audit-org_1`），原因：

- **生命周期管理**：金融监管通常要求操作记录保留 N 年。按月分 bucket 后，过期数据的清理粒度为月级别，直接删除整个 bucket 即可，无需遍历删除
- **容量均衡**：按组织分可能导致大型组织的 bucket 迅速膨胀，按月分则每个 bucket 的增长是有界的
- **跨组织隔离不依赖 bucket**：组织级数据隔离由 object key 路径（`audit/{org_id}/...`）保障，而非 bucket 边界

### 脱敏规则的执行顺序

4 条规则按特定顺序执行：password → card_number → id_number → phone。password 规则最先执行，因为密码字段可能包含看似卡号或手机号的字符串（如 "password: 6222021234561234"），若卡号规则先执行会将密码中的数字部分脱敏但保留末 4 位——这不符合密码"完全掩码"的要求。

### 审计写入的降级处理

`write_audit_log()` 在写入失败时的行为是"告警但不中断"：

- 记录 `AUDIT_LOG_FAILURE` 级别的日志，包含完整的错误上下文（task_id、action_index、异常信息）
- 回滚当前数据库事务（防止脏状态影响后续操作）
- 返回 None 而非抛出异常

在生产环境中，`AUDIT_LOG_FAILURE` 应触发运维告警（通过日志采集系统的关键词告警规则），确保审计缺失被及时发现和补录。

## 金融企业场景下的工程意义

**监管合规留存**：银保监会《银行业金融机构数据治理指引》要求"建立数据安全技术标准，采取有效措施保障数据安全"。全链路审计日志完整记录了 RPA 系统的每一步操作，包括操作前后的页面截图，为监管检查提供了可审计、可追溯的操作证据链。

**操作回溯能力**：当出现"谁在什么时间对哪个账户做了什么操作"的追溯需求时，审计日志提供了完整的操作时间线。每条日志包含操作前后截图，可以精确重建操作现场，这在处理客户投诉、内部审计、监管问询等场景中至关重要。

**敏感信息保护**：RPA 操作过程中不可避免地接触银行卡号、身份证号、密码等敏感信息。脱敏引擎在数据写入审计日志前自动处理，确保敏感信息不在日志系统中明文存储。SHA-256 哈希保留了验证能力但不可逆向还原。

**截图的时效性访问控制**：审计截图通过 MinIO 预签名 URL 提供临时访问，默认 1 小时有效期。这避免了长期有效的截图链接在通讯记录、邮件等渠道中被转发和滥用，符合最小权限原则。

## 踩坑记录

1. **脱敏规则的正则冲突**：身份证号（18 位数字+X）和银行卡号（16 位纯数字）的正则存在长度重叠区间。通过限定银行卡号的首位为 3-6（主流发卡行 BIN 范围）和身份证号的出生年份前缀（19/20）来区分，避免了误匹配
2. **密码字段的关键词覆盖**：中国金融系统的界面可能使用"密码""口令""passwd""pwd"等多种表述。初始仅覆盖 `password`，补充后覆盖了 5 种常见变体，通过 `re.IGNORECASE` 同时兼容大小写
